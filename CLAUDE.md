# bushglue

## Agent contribution norms

This repo is maintained by one meatbag at a time. Agent contributions land here from multiple sources. To keep the signal-to-decoration ratio high:

- **Default to updating existing docs only as necessary.** If the change is self-evident from the diff + commit message, don't write a 1-pager. Focus on user-facing critical info only.
- **One doc per change, not a fan-out.** No separate 1-pager + walkthrough + full review + amendment for the same work. Pick one.
- **Don't invent decisions for a fictitious team.** There is one meatbag reading. Address them directly. No "the team needs to decide X" sections, no "Ask:" closers, no rhetorical decision lists.
- **Risk/severity tables** only when there are more than three real risks. Two risks fit in two sentences.
- **Write to the operator, not to a stakeholder.** Skip the executive-summary register. Plain technical prose.

## Valve control — where it stands (2026-06-10)

On `main`: the valve node is back on **CAN** — a small fleet, all CircuitPython sharing one `valve.py` CAN driver (the SERVO42D closed-loop driver, restored from `6e1c03f`), differing only in host link + CAN carrier. Three node types: **Pico 2 W + CanBerry MCP2515** over Wi-Fi/MQTT (`code.py`, `BOARD="canberry"`, SPI GP6/7/4/5, 16 MHz; now also takes **multiple Wi-Fi networks** `NETWORKS` + **multiple brokers** `MQTT_BROKERS`, tried before the /24 scan); **XIAO nRF52840 + MCP2515** over BLE → `bush_valve_ble` (`code_xiao_ble.py`); and **Waveshare RP2350-CAN** (onboard XL2515, no radio) over **USB-serial → the new `bush_valve_serial` bridge** (`code_usb_serial.py` + `boot.py`). Still deferred (no hardware): a reflashed **BridgePlate + CanBerry** centerpiece, also USB-serial (Pi-Plates coexistence rules in the plan). Motor link is CAN-only; UART and STEP/DIR are retired (recover step/dir from `3f40ee8`, UART from `9c709f5^`). 42D menu: CAN 500 kbps, ID 1, MStep 16 — the firmware sets SR_vFOC + run current over the bus (there is no "CR_CAN" work mode). Protocol + host-link table in `firmware/valve-control/PROTOCOL.md`.

**Bench status (2026-06-10): full CAN closed loop PROVEN with the 42D motor on the Waveshare node.** `0x31` encoder reads (live during motion AND while stalled), `init()` ACKs, motion, and gentle seat homing all work. The bring-up blocker was **CAN bus termination** — with no 120 Ω terminator the XL2515 can't even transmit (TXREQ stuck, TXERR=1, TEC frozen at 0 — distinct from the no-ACK/TEC-climb case); one terminator (the 42D end) fixed it instantly. Verified + locked into `valve.py`: **`DIR_TOWARD_OPEN=0x00` / `DIR_TOWARD_CLOSED=0x80`** (the old guesses were inverted), `_enc_sign=-1`; **current = torque** (`0x83`, the only knob — no separate torque cmd on the 42D), `VALVE_CURRENT_MA=1000` for fast moves; **lockrotor protection kept ON** (`init` sends PROTECT `0x01`) as the seat detector (`0x3E` latch) + operating jam net — it does NOT false-trip at adequate current, and `0xF1` gives run status; **full travel ~3.76 rev, `OPEN_STEPS=11200`**. (Flashed CP 10.2.1 `raspberry_pi_pico2`; XL2515 16 MHz; SIT65HVD230 3.3 V.) The **Pico 2 W + CanBerry and XIAO** builds gave bench trouble and are NOT yet proven. CircuitPython gotchas fixed: no `del` on `bytearray` (use `_rx[:] = _rx[n:]`); writing `code.py` auto-reloads AND a manual Ctrl-D reloads = **double run** (enter the REPL first for one-shot motion probes).

**Homing is ENABLED + bench-proven, and deliberately GENTLE** (`HOMING_DISABLED=False`): `cmd_home()` blocking protection-seek — drive toward the closed seat at LOW current (300 mA), the 42D latches `0x3E` at the seat, back off a margin, zero. **It must NEVER bump current to force past resistance — forcing shreds the valve internals** (that grinding made the current bench valve's sticky spots), so rough homing perpetuates the damage; a worn valve errors `home_stuck` instead of being pushed. The **current bench valve is too chewed to home gently** (moves only when forced to 800–1000 mA) → it needs replacing; do NOT raise the homing current to compensate. Safety asymmetry: the CLOSED seat took 800 mA fine (~1 microstep) but the **OPEN hard stop JAMS at high current** — operation stays within `[0, OPEN_STEPS]` so it never rams either. Next homing avenue: the 42D's own internal homing (`0x90`/`0x91`/`0x92`, watch `0xF1`/`0x3E`; mind the NVM auto-home-rams-on-boot gotcha). More in the MKS-quirks and closed-seat memories.
